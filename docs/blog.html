Why neural games ignore your buttons

a negative result, and the mechanism behind it · 2026-07

We built two tiny games — a run-and-gun platformer and a pseudo-3D racer — specifically to train neural networks to become the game engine: a model that generates each frame from pixels, the player's input, and its own memory, with no simulator in the loop. Playable originals are here. This post is about what went wrong, because what went wrong is more interesting than a demo.

The model learned to generate convincing gameplay footage. It also learned to completely ignore which button you pressed. NO-GO

The setup nobody warns you about

Our games are deterministic simulators with a gift: from any moment we can fork the exact same state down many different button-presses and record the true futures. That gives perfectly action-labelled data and paired counterfactuals — the dream dataset for learning action-conditioned dynamics. We trained a small (≈9–13M) tokenizer + block-causal dynamics model on ~700k frames, the standard MaskGIT recipe with the action injected as conditioning.

Then we measured whether the trained model actually responds to actions. It does not.

Two independent measurements, same verdict

Real transitions carry an action signal, but almost all of the recoverable "0.63" is the copy-last static prior — the genuine motion signal is only ≈0.03. In the model's own generations, actions are unrecoverable (AUC 0.49) and different actions produce bit-identical futures (separation 0.0).

The mechanism: action = a shift that softmax throws away

We instrumented the model on real hardware. Feed it the same context and two different actions and watch the output logits over the token codebook:

quantityvaluereading
logit max-difference (2 actions)1.06 – 1.46the action does move the logits
softmax max-difference2.8e-15…but the probability distribution is unchanged
decoded pixel difference0.0so the frame is identical
The action's effect is a nearly-uniform additive shift across the codebook. Adding a constant to every logit leaves the softmax exactly where it was. The model technically "uses" the action and functionally ignores it.

Why it happens — and why it's hard to fix

The disease is the objective. At the game's frame rate, consecutive frames barely change, so the reconstruction loss saturates to near-zero (CE ≈ 5e-5) using context alone. There is simply no gradient pressure to use the action — predicting the next frame without it is already almost perfect. The action pathway atrophies.

We tested this directly. We gave the model the architectural capacity to be action-discriminative: a low-rank action → codebook logit bias that, on an untrained network, moves the softmax by ~1e-2 (twelve orders of magnitude above the degenerate baseline). After training on reconstruction alone, it collapsed back to ~2.5e-7. The capacity was there; the objective trained it away.

This is the "action marginalization" failure mode of generative world models — but with a clean, measured mechanism: reconstruction saturation drives the action signal to zero, and mere architectural capacity to use actions does not survive it.

What we're doing about it

The remedy has to change the objective, not just the architecture. Two levers:

① Counterfactual / branch-consistency supervision. Because our simulator gives paired futures for the same state under different actions, we can supervise the model directly: inside the action's region of effect, reproduce the real branch difference; outside it, stay invariant. This gives the gradient that reconstruction never will. Getting its gradient to actually flow took fixing two silent bugs (a masked-CE that returned exactly zero; a teacher-forcing decode that washed out the action) — documented, with regression tests.

② Make actions matter per step. Predict frames further apart so a single action visibly changes the frame, breaking reconstruction saturation.

That work is ongoing and honest about its state: architecture and objective are fixed and verified; whether the combination revives measurable controllability is the open question we're running now. Whatever the answer, the finding above stands on its own — and it's the kind of thing that only shows up when you refuse to accept "it looks like the game" as success.
